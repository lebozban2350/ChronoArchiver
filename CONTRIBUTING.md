# Contributing

## Development setup

- Python **3.10+**
- From the repo root: `python src/bootstrap.py` (creates the app-private venv under the platform user data directory)
- Optional: `pip install -e ".[dev]"` for pytest, **pre-commit**, and **mypy** in an isolated environment
- Optional hooks: `pre-commit install` then `pre-commit run --all-files` (Ruff check + format; same rules as CI)

Run tests with `PYTHONPATH=src` or `pytest` (see `pyproject.toml` `pythonpath`).

### Type checking

- **`mypy src/core`** uses **`[tool.mypy]`** in `pyproject.toml`. CI runs this on **`src/core`** as an **advisory** step until the tree is fully clean.
- Prefer explicit hints on new or touched public APIs in **`core/`**.

### First-run / upgrade UX (reference)

- After a version bump, users may see a **what’s new** dialog once; the last seen version is stored under **Settings** / `last_seen_app_version.txt`. **Do not show** is stored in **Qt** settings. Respect these when testing upgrade flows.

### Performance profiling (maintainers)

- For hot paths (encode, upscale, scan), use standard tools (**`cProfile`**, **`py-spy`**, or your OS profiler) on representative inputs. Do not commit large traces or flame graphs into the repo.

## Release checklist (maintainers)

1. Merge work to `main` and ensure **CI** is green.
2. Bump the SemVer across the canonical files from the repo root:
   ```bash
   python tools/bump_version.py X.Y.Z
   ```
   This updates `src/version.py`, `pyproject.toml`, `README.md`, `PKGBUILD`, and installer-related defaults.
3. Update `CHANGELOG.md` with a dated entry for the new version (follow existing structure).
4. Commit with a conventional message, e.g. `chore: release X.Y.Z`.
5. Tag and push: `git tag vX.Y.Z` and `git push origin main --tags`.
6. Build or publish installers and the **AUR** package as needed; align `pkgver` with the tag (handled by `bump_version.py` for `PKGBUILD`).
7. For GitHub **Releases**, publish **SHA-256 checksums** (or the checksum files generated by the build) alongside installers so users can verify downloads. When your build pipeline supports it, attach **provenance** or **SLSA**/GitHub **artifact attestation** metadata so consumers can verify the supply chain for that release.
8. **Windows / macOS installers:** sign binaries with your code-signing identity where applicable; on macOS, **notarize** the app bundle for Gatekeeper. Document the signing pipeline in internal release notes if non-obvious.

## CI expectations

- **pytest** runs under `PYTHONPATH=src` with optional PyTorch for Real-ESRGAN import tests; includes an offscreen **main window** smoke test when `CHRONOARCHIVER_CI=1`.
- **Ruff** lint (`ruff check`) and **format** (`ruff format --check`) on `src/` and `tests/` — both must pass.
- **Mypy** on `src/core` runs in CI for visibility only (`continue-on-error: true`) until coverage is complete.
- **Version sync**: `src/version.py` `__version__` must match `pyproject.toml` `[project].version`.
- **Dependabot** opens weekly PRs for `requirements.txt`; **dev** dependencies are grouped (`pytest`, `ruff`). **torch** / **torchvision** / **torchaudio** are ignored (bump manually with testing).

## Diagnostics and privacy

The app can **export a local ZIP** (environment + log tail) to a path **the user picks** (e.g. Desktop or Downloads). **Nothing is transmitted** to ChronoArchiver or any third party unless the user manually attaches that file to GitHub or email. There is **no** built-in “phone home” or anonymous server endpoint. See [SECURITY.md](SECURITY.md).
